# UI 方向实验室

这个 skill 用于进入编码前的视觉方向探索。它不绑定具体项目，适用于 Web、H5、App、后台工具、设计系统示例、活动页、功能模块等界面。目标是先理解项目已有风格和设计师偏好，再和设计师 QA 对齐，最后生成多张独立参考图并持续讨论收敛。

## 核心原则

- 把用户当作设计师沟通：技术判断要翻译成设计语言、产品目标和实现影响。
- 先读项目，再做风格判断；不要凭通用审美覆盖项目既有风格。
- 生成图片前必须 QA 对齐方向和张数，不能直接生图。
- 默认先探索多个视觉方向；除非用户明确要求实现，否则不要直接进入写代码。
- 如果可以选择模型，视觉推理、方向推演和提示词细化阶段优先使用 GPT-5.5。
- 如果当前环境可用图像生成能力，要生成真实参考图；如果不可用，要提供可直接执行的图像提示词，并说明当前环境无法直接生成图片。

## 1. 读取项目风格和设计师倾向

执行前先按当前项目实际情况轻量读取相关资料，避免过度扫描：

- 项目规则：`AGENTS.md`、`AI_RULES.md`、`.cursorrules`、`CLAUDE.md`、`README.md` 中和设计/实现有关的规则。
- 设计系统：tokens、theme、component registry、DesignSystem 页面、Storybook、example manifest、公共组件示例。
- 现有页面：与本次需求同类型或同业务域的 2-4 个代表页面。
- 设计师倾向：用户当前诉求、最近反馈、明确偏好的关键词、参考图中可复用的设计特质。

只提炼和本次设计有关的结论，例如：色彩倾向、圆角/阴影/留白、信息密度、动效强度、插画/图像使用方式、品牌或业务禁忌。

## 2. 诉求解析

- 判断目标界面类型：页面、模块、卡片、弹窗、引导、任务流程、报告页、表单、列表、仪表盘或空状态。
- 提取目标用户、核心任务、情绪目标、信息密度、关键操作、平台尺寸和隐私/合规约束。
- 如果用户提供参考图，只提炼可借鉴的设计特质，不直接复制。
- 如果是高价值实现任务，提前判断哪些方向适合落地，哪些只适合概念探索。

## 3. 生成前 QA 对齐

在生图前必须和设计师做一次简短 QA。问题要少而关键，优先一次性问完：

- 本次更偏“探索风格”还是“接近可落地稿”？
- 希望生成几张独立参考图？
- 是否要覆盖多个方向，还是围绕一个方向做多个变体？
- 是否有必须保留或必须避开的视觉元素、品牌限制、竞品参考？

如果用户已经明确给出这些信息，可以不重复问，只用一句话复述确认：“我将按 X 个方向生成 Y 张独立图，每次生图只出一张完整界面。”

## 4. 提出方向

通常提出 3-5 个差异明显的方向。每个方向包含：

- 方向名称
- 视觉情绪
- 布局结构
- 关键交互或动效想法
- 色彩和材质策略
- 适合场景
- 主要风险

方向必须来自“项目风格事实 + 设计师倾向 + 当前功能目标”的交集，而不是只换颜色。

## 5. 生成参考图

- 必须按 QA 结果生成指定张数。
- 多张参考图必须是多次独立生图请求：每次请求只生成一张图，只对应一个方向或一个变体。
- 每张图只展示一个完整界面，不要把多个界面、多个方案、多张手机稿或多张桌面稿拼在同一张图里。
- 根据项目平台选择画幅：移动 H5/App 常用 9:16；桌面工具常用 16:10 或 1440px 宽截图；组件/模块可用聚焦局部构图。
- 每张图都应像真实产品 UI 参考，而不是海报、落地页或抽象情绪板，除非用户明确要做营销视觉。
- 可以加入真实中文 UI 标签，但如果图像模型容易扭曲文字，应降低文字密度，把重点放在布局、层级和风格。
- 调用图像生成前，先说明计划生成的方向和张数。若图像工具要求生成后保持静默，调用后不要继续补充说明。

## 6. 讨论并收敛

- 引导设计师从感觉、清晰度、新鲜感、项目适配度、可落地性几个维度比较。
- 用户反馈后，保留最强特质，只修正薄弱部分。
- 可以围绕被选中的方向再生成 2-3 张更聚焦的变体，但仍然必须多次独立请求生图接口。
- 收敛后输出“方向决策摘要”：保留什么、放弃什么、下一步如何落地。

## 7. 实现交接

当用户选定方向并要求落地时：

- 把视觉方向翻译成实现 brief：层级、组件、tokens、状态、动效、资产、文案和响应式要求。
- 优先复用当前项目已有设计系统、组件和 token；只有现有资产无法覆盖时才建议新增。
- 如果项目规则要求截图或渲染验证，UI 代码改动后必须执行，并持续调整到目标效果真实可见。

## 参考图提示词模板

按下面结构生成，并结合项目和任务改写：

```text
[项目/产品名] 的 [平台类型] UI 截图，[画幅]。
只生成一张独立界面图：画面中只有一个完整界面、一个方案，不要多屏拼图、不要四宫格、不要多个截图并排。
功能：[具体页面、模块或流程]。
方向：[方向名称和情绪]。
项目风格依据：[从项目规则、设计系统、现有页面中提炼的风格结论]。
设计师倾向：[用户明确偏好或参考图特质]。
布局：[顶部/中部/底部或主区/侧栏/工具栏结构]。
视觉风格：[现代、高级、有趣等具体化描述，避免泛词堆叠]。
内容：[真实但隐私安全的 UI 信息，清晰主操作]。
交互暗示：[动效或状态提示]。
避免：[项目禁忌、用户不喜欢的风格、图像模型常见错误]。
```

## 质量标准

- 生成前已读取项目风格事实和设计师倾向。
- 生成前已 QA 对齐方向、变体策略和图片张数。
- 每张图都是单独请求生成的独立界面，不是拼图。
- 每个方向必须有明显差异，不能只是换颜色。
- 首屏应马上让用户知道能做什么。
- 视觉亮点要帮助理解，而不是抢走注意力。
- 如果某个方向不适合当前项目实现，要在推荐前明确说明。
